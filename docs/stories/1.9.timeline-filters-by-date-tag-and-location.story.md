## Story

**As a** user,
**I want** to filter the memory timeline,
**so that** I can find records without scrolling through every memory.

## Acceptance Criteria

1. Users can filter records by date or date range.
2. Users can filter records by tag.
3. Users can filter records by location.
4. Filters can be cleared to restore the full tenant-scoped timeline.
5. Filters do not turn the timeline into a table or admin list.
6. Filter labels and states are safe for English, Portuguese, and Spanish strings.

## Integration Verification

1. **IV1:** Filters never return records outside the current tenant.
2. **IV2:** Filter queries eager load needed record relationships to avoid avoidable N+1 behavior.
3. **IV3:** Tests verify filtered results, clearing filters, and tenant scoping.

## Implementation Interpretation

Story 1.9 adds focused filtering to the existing private, month-grouped memory timeline. Diary records already use `experience_date`; trip/period records already use `period_start_date`; timeline cards already render date, location, tags, excerpts, photo counts, and record-detail navigation. This story should make those existing timeline signals filterable without changing the product into a table, admin list, report, map, masonry view, or advanced search surface.

The preferred implementation is to keep filter state in `App\Livewire\Memory\MemoryTimeline`, delegate tenant-scoped query construction to `App\Services\MemoryTimelineService`, and keep rendering in `resources/views/livewire/memory/memory-timeline.blade.php`. Date and selected tag may use URL-backed Livewire state where useful. Location search must remain transient by default because it is free text and can be privacy-sensitive.

## Scope Boundaries

- Add filters only for date/date range, tag, and location on the memory timeline.
- Preserve the vertical month-grouped timeline, existing cards, detail links, photo count behavior, and current diary/period ordering.
- Keep all query logic tenant-scoped inside `MemoryTimelineService`; Blade must not query private records, tags, media, or highlights.
- Use existing tenant-scoped `MemoryTag` records for tag filter options.
- Date filters may include a single date or a from/to range as long as diary and period records use their proper timeline date.
- Location filtering should match the user-entered `location_name` field and stay out of the URL by default.
- Clearing filters must restore the full current-tenant timeline and reset active filter labels.
- Add no public sharing, visibility toggles, analytics dashboards, validation reporting, album language, map browsing, masonry layout, or admin table framing.
- Do not change account, billing, dashboard, invitation, workspace-management, or private media delivery behavior beyond regression preservation.

### Source Documents

- PRD Story 1.9 defines the user story, six acceptance criteria, and three integration verification points for timeline filtering by date, tag, and location. [Source: docs/prd.md#story-1.9-timeline-filters-by-date-tag-and-location]

- Architecture assigns timeline filtering to `MemoryTimelineService`, including date, tag, and location filters, ordered month grouping, and eager-loaded tags and media. [Source: docs/architecture.md#service-architecture]
- Architecture uses tenant-scoped `memory_tags` and a pivot table so filtering stays simple and efficient. [Source: docs/architecture.md#memory_tags-and-memory_record_tag]
- UI architecture shows the intended Livewire state model: date filters and selected tag may be URL-backed, while location remains transient by default. [Source: docs/ui-architecture.md#state-management]

- Framework coding standards require tenant boundaries before rendering private content, service-owned timeline queries, privacy-safe URL state, localization keys, and no Filament/table/admin framing for memory workflows. [Source: docs/framework/coding-standards.md#memory-product-rules]
