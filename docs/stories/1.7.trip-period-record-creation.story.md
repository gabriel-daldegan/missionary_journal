## Story

**As a** user,
**I want** to create a trip or period record,
**so that** multi-day memories can appear naturally on the same timeline.

## Acceptance Criteria

1. Users can create trip/period records with title, location, start date, end date, people, notes, and Memory Highlights.
2. The end date must be on or after the start date.
3. Trip/period records appear on the timeline by period start date.
4. Trip/period records remain private to the tenant/workspace.
5. The form stays structured and faster than a long-form diary entry.

## Integration Verification

1. **IV1:** Trip/period records share the memory timeline ordering model without disrupting diary records.
2. **IV2:** Unauthorized users cannot create or view trip/period records in another tenant.
3. **IV3:** Tests cover successful creation, date validation, timeline placement, and cross-tenant denial.

## Implementation Interpretation

Story 1.7 activates the reserved `period` memory record type for creation. The existing `memory_records` schema already contains the period fields and Story 1.6 already made `MemoryTimelineService` timeline-date aware for period records. This story should therefore reuse the current tenant-safe route stack, `MemoryRecordEditor`, `MemoryRecordService`, tags, ordered Memory Highlights, detail route, and timeline service rather than introducing a second record subsystem.

The scope is creation only. Period edit polish can remain out of scope unless a small shared editor adjustment is required to avoid breaking current diary edit behavior. The implementation must not add private media upload or delivery, filters, maps, masonry, complex animation, public sharing, visibility selectors, album-generation copy, or account/dashboard changes.

## Scope Boundaries

- Activate `period` only as a supported create type in the existing memory record route/editor flow.
- Reuse `MemoryRecord::TYPE_PERIOD`, `period_start_date`, `period_end_date`, `title`, `notes`, `location_name`, `people`, tags, and ordered highlights from the existing schema.
- Keep diary creation and diary edit behavior stable.
- Keep the period form structured and faster than a long-form diary entry: title, location, start date, end date, people, notes, and highlights should lead the flow.
- Do not implement period edit as a separate feature unless the shared component needs minimal type-safe handling for existing routes.
- Do not add private media upload, media delivery, filters, public sharing, per-record visibility modes, masonry, map browsing, complex timeline animation, or album-generation language.
- Do not change account, billing, dashboard, invitation, or workspace-management flows unless required for regression fixes.

### Architecture Guidance

- Memory product routes live under `/memories/{tenant:uuid}` with `memories.records.create`, `memories.records.show`, and `memories.records.edit` as customer-facing Livewire/Blade screens, not Filament resources. [Source: docs/architecture.md#product-routing-and-ui-architecture]
- The MVP timeline is a vertical month-grouped list, and `MemoryTimelineService` should order records by experience date or period start. [Source: docs/architecture.md#product-routing-and-ui-architecture]
- `memory_records` stores diary and trip/period records on the same tenant timeline with `type`, `title`, `body`, `notes`, `experience_date`, `period_start_date`, `period_end_date`, `location_name`, and `people`. [Source: docs/architecture.md#data-model]
- Period records require `title`, `period_start_date`, and `period_end_date`; `period_end_date` must be on or after `period_start_date`. [Source: docs/architecture.md#data-model]
- `MemoryRecordService` owns record create/update, tag sync, ordered highlight sync, and tenant-scoped writes. [Source: docs/architecture.md#service-architecture]
- Create actions should route explicitly to supported record types `diary` or `period`; route-bound records must be re-scoped to the route tenant before rendering. [Source: docs/ui-architecture.md#route-inventory; docs/ui-architecture.md#navigation-rules]
- Record editor tests should cover required validation, diary creation, trip/period creation, highlight persistence, shared editing, and unauthorized create/update denial. [Source: docs/ui-architecture.md#route-tests-required-by-stories; docs/ui-architecture.md#test-coverage-matrix]
- The period form should lead with title, location, and dates while keeping optional metadata secondary. [Source: docs/framework/coding-standards.md#memory-product-rules]
- Customer-facing memory UI belongs in Livewire/Blade memory screens; tenant and record authorization belongs in middleware, policies, services, and scoped Eloquent queries before private content reaches a view. [Source: docs/framework/source-tree.md#file-placement-rules]
