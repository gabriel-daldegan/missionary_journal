## Story

**As a** product team,
**I want** lightweight validation reporting and a runbook,
**so that** MVP decisions are based on real usage without exposing private memory content.

## Acceptance Criteria

1. Admin-only reporting can identify profile completion, first record creation, and active validation tenants.
2. Reporting can identify tenant activity across more than two weeks.
3. Reporting can support family/collaborator engagement validation.
4. Optional email-source behavior can be measured if the non-blocking marker is included.
5. Reporting does not expose private record body content to workspace users.
6. The MVP runbook covers registration, profile setup, language selection, diary creation, trip/period creation, photo attachment, timeline browsing, filters, tenant access, family workspace access, and shared editing.
7. The runbook separates MVP validation from post-MVP feature requests.

## Integration Verification

1. **IV1:** Validation reporting is available only to admins or internal operators.
2. **IV2:** Workspace users cannot access validation dashboards, metrics, or behavior summaries.
3. **IV3:** Tests or documented checks verify admin-only access and absence of private record body content in validation events.

## Implementation Interpretation

Story 1.11 closes the Epic 1 MVP validation loop. It should provide a small operator-only way to answer whether validation tenants are completing profiles, creating first records, staying active over more than two weeks, engaging collaborators/family members, attaching media, and optionally creating email-sourced records. The implementation should prefer derived metrics from existing memory data before adding persistence. A small `memory_activity_events` table is acceptable only if derived records, profiles, tenant membership, timestamps, media, and `MemoryRecord::source` / `source_metadata` cannot satisfy the reporting need. [Source: docs/architecture.md#validation-reporting-architecture]

The required reporting surface is a small Filament Admin page because the app already discovers admin pages under `app/Filament/Admin/Pages`, the PRD calls for admin-only reporting, and the existing admin panel already gates non-admin users. An internal Artisan command/report is optional only if Dev determines it materially reduces validation work; if added, it must reuse the same metric service and include focused command-output tests. [Source: docs/architecture.md#validation-reporting-architecture] [Source: docs/framework/source-tree.md#existing-application-areas]

The MVP runbook is a required deliverable, not a code comment. Create `docs/validation/mvp-runbook.md` and keep it focused on validation workflows, operator checks, and post-MVP request capture. It must not turn post-MVP requests into success criteria for this story. [Source: docs/prd.md#story-1.11-admin-only-validation-reporting-and-mvp-runbook]

## Scope Boundaries

- Build the smallest admin-only reporting mechanism that satisfies AC1-AC5.
- Prefer query-derived metrics from `memory_profiles`, `memory_records`, `tenant_user`, media, workspace settings, and existing tenant/user relationships.
- Add `memory_activity_events` only if a specific required signal cannot be derived safely from existing data.
- If event persistence is added, store safe event names and metadata only; never store private record body, notes, highlight text, location free text, filenames, media paths, or tenant-private notes.
- Keep reports operator-facing. Do not expose validation dashboards, metrics, behavior summaries, or runbook-only data to workspace users.
- Keep customer-facing memory UI unchanged. Do not rebuild timeline, filters, record editor, media delivery, localization, profile gate, tenant access, invitation, billing, or account behavior.
- Do not build public analytics, customer-facing dashboards, product-wide BI, AI analysis, automated email ingestion, or post-MVP feature tracking.
- Do not translate or summarize user-generated content.
- Do not make the runbook a product surface. It is documentation for MVP validation execution.
