## Story

**As a** workspace member,
**I want** to edit and open diary records,
**so that** memories can be corrected, refined, and read in full after capture.

## Acceptance Criteria

1. Authorized workspace members can open a diary record detail view.
2. Authorized workspace members can edit body text, experience date, location, tags, and Memory Highlights.
3. Editing preserves record identity.
4. Timeline placement updates when the experience date changes.
5. Unauthorized users cannot view or edit records outside their tenant.
6. Detail views do not display missionary-only or family-visible labels.

## Integration Verification

1. **IV1:** Existing account/dashboard operations remain unaffected by record editing.
2. **IV2:** Cross-tenant record detail and edit requests are denied without leaking record content.
3. **IV3:** Tests cover authorized view/edit, unauthorized denial, and timeline date updates.

## Implementation Interpretation

Story 1.5 is the next scope after Story 1.4c and starts detail/edit on top of the completed diary schema, service/policy, and create UI foundations. It adds: record detail read page and record edit flow in the memory product. It must preserve tenant-scoped access and must continue to avoid visibility-mode selectors, period/trip workflows, media delivery changes, public sharing, AI classification, and admin-table style screens.

The timeline remains outside this story except for exposing record detail navigation from existing timeline records where applicable.

## Scope Boundaries

- Use Story 1.4's completed schema/service/UI baseline.
- Do not introduce per-record visibility controls (`missionary-only`, `family-visible`).
- Do not implement trip/period create or edit flows.
- Do not add media upload/media delivery behavior.
- Do not change profile gating, invitation flows, or account/Dashboard operations.
- Do not use bypass routes or route-level exceptions to avoid tenant checks.

### Code Quality Assessment

PASS for Story 1.5 after architect review. The implementation adds tenant-scoped detail and edit routes, record detail rendering, create/edit reuse in `MemoryRecordEditor`, transactional service updates, policy-backed view/update authorization, timeline detail links, and date-based timeline ordering without introducing forbidden visibility, media, trip/period, public-sharing, or AI/source controls.

Architect review found one real route regression before issuing the gate: `memories.records.edit` returned a generic 404 for an authorized member because the route `{record}` parameter was being interpreted through the create-route `$type` mount argument. I fixed the mount contract to let Livewire inject the typed public `record` property for edit mode while keeping the create `{type}` guard explicit, then added positive HTTP coverage for the edit route.

### Refactoring Performed

- **File**: `app/Livewire/Memory/MemoryRecordEditor.php`
  - **Change**: Simplified `mount()` to accept only `Tenant $tenant` and the create-route `string $type`, and to enter edit mode from the existing typed public `$record` property.
  - **Why**: The previous union parameter mixed create and edit route semantics and caused authorized edit GET requests to abort before rendering.
  - **How**: Aligns the component with Livewire route parameter/property resolution while preserving unsupported create-type rejection.

- **File**: `tests/Feature/Livewire/Memory/MemoryRecordEditorTest.php`
  - **Change**: Added `test_member_with_completed_profile_can_access_diary_edit_route`.
  - **Why**: Existing tests covered direct component edit mode and negative edit-route access, but not the positive HTTP edit route.
  - **How**: Proves the route renders edit copy and prefilled body/location/highlight content for an authorized completed-profile member.
