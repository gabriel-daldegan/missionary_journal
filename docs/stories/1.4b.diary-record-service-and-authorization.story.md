## Story

**As a** workspace member,
**I want** diary record creation to go through a tenant-scoped service and authorization boundary,
**so that** records, tags, and highlights are persisted safely before the UI depends on them.

## Acceptance Criteria

1. Authorized tenant members can create diary records through `MemoryRecordService`.
2. Diary creation writes `tenant_id`, `author_user_id`, `last_edited_by_user_id`, `type = diary`, `body`, `experience_date`, and optional `location_name`.
3. Tags are created or reused within the current tenant and never shared across tenants.
4. Memory Highlights are persisted as ordered structured rows for the created record.
5. Users outside the tenant are denied before any record, tag, or highlight write.
6. Service/policy code does not expose missionary-only, family-visible, collaboration mode selector, media, period/trip, detail/editing, or public sharing behavior.

## Integration Verification

1. **IV1:** Story 1.4a schema/model tests remain passing.
2. **IV2:** Existing Story 1.3 shared-editing decisions remain the source of create authorization.
3. **IV3:** Service/policy tests cover successful creation, tag reuse/isolation, highlight ordering, required validation assumptions, and unauthorized tenant denial.

## Implementation Interpretation

This is the second slice of PRD Story 1.4. It requires Story 1.4a. It adds only the backend service and policy boundary for diary creation. Do not add the Livewire editor, create route, Blade form, translations, detail route, edit route, timeline grouping, filters, or media behavior in Story 1.4b.

### Architecture Guidance

- Access rejects users whose `User::canAccessTenant($tenant)` check fails and uses policy checks for memory access. [Source: docs/architecture.md#tenant-and-access-architecture]
- Authorize create and update through policies/services even while both MVP roles can edit. [Source: docs/architecture.md#tenant-and-access-architecture]
- `MemoryRecordService` creates and updates records, syncs tags, syncs ordered highlights, and enforces tenant-scoped writes. [Source: docs/architecture.md#service-architecture]
- Livewire components should call services rather than building write logic directly in views; this story creates that service boundary for later UI. [Source: docs/architecture.md#service-architecture]
- The UI architecture service template shows the intended create method shape and helper responsibilities, but final code must use the schema field names from `memory_records`. [Source: docs/ui-architecture.md#service-template]
