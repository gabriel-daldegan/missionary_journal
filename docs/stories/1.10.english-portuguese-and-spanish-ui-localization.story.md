## Story

**As a** user,
**I want** to use the memory product in my preferred supported language,
**so that** profile setup, capture, and timeline browsing are understandable.

## Acceptance Criteria

1. Memory product UI strings are available in English, Portuguese, and Spanish.
2. English is the default interface language.
3. Users can select their preferred language during memory profile setup and change it later from profile-related controls.
4. User-generated content remains in the language entered by the user.
5. Core forms, buttons, filters, cards, navigation, empty states, and access-denied states fit supported languages.
6. Localization does not introduce content translation behavior.

## Integration Verification

1. **IV1:** Existing Laravel localization conventions are used.
2. **IV2:** Existing account/dashboard localization behavior is not broken by memory-specific locale handling.
3. **IV3:** Tests cover locale preference affecting memory UI strings without translating record content.

## Implementation Interpretation

Story 1.10 completes the memory-product localization slice on top of the existing profile setup, diary/period editor, record detail, timeline filters, private media, and layout shell. `MemoryProfileService` already defines `en`, `pt`, and `es`; `memory_profiles.preferred_locale` already exists; `MemoryProfileSetup` already captures a preferred locale; and `resources/lang/en|pt|es/memory.php` already contain substantial memory UI strings. This story should finish the locale wiring, fill translation-key gaps, and add regression coverage instead of rebuilding the profile model or introducing a new language framework.

The preferred implementation is to keep Laravel localization as the only UI localization mechanism, set the authenticated memory product surface locale from `MemoryProfileService::preferredLocale()`, keep English as the default/fallback, and ensure every customer-facing memory string uses `__('memory.*')` keys. User-generated record content, tags, locations, people, highlights, filenames, notes, and mission context must render exactly as entered.

## Scope Boundaries

- Localize customer-facing memory product UI for English, Portuguese, and Spanish.
- Keep supported locales constrained to `en`, `pt`, and `es`.
- Keep English as default/fallback for missing or unsupported profile locale values.
- Preserve existing account, billing, dashboard, invitation, tenant membership, route authorization, profile gate, timeline, record editor, record detail, filter, and private-media behavior.
- Use existing Laravel localization conventions and existing `resources/lang/{locale}/memory.php` files in this repo.
- Do not add automatic translation, AI translation, machine translation, per-record language detection, browser-locale negotiation, public sharing, album-generation copy, admin localization, analytics dashboards, validation reporting, or a new frontend framework.
- Do not translate user-generated title, body, notes, tags, locations, people, highlights, filenames, media metadata, mission context, tenant names, or display names.
- Admin-only screens may remain English unless they are tester-facing in the memory validation flow.

### Current Implementation Context

- `App\Services\MemoryProfileService` already defines `DEFAULT_LOCALE = 'en'`, `SUPPORTED_LOCALES = ['en', 'pt', 'es']`, `preferredLocale()`, `normalizeLocale()`, and profile completion behavior.
- `App\Livewire\Memory\MemoryProfileSetup` already captures `preferredLocale`, validates it against `MemoryProfileService::SUPPORTED_LOCALES`, sets locale during `mount()`, and saves the preference through `MemoryProfileService`.
- `App\Http\Middleware\EnsureMemoryProfileComplete` already calls `app()->setLocale($this->memoryProfileService->preferredLocale($user))` for routes behind `memory.profile.complete`.
- Memory routes live under `/memories/{tenant:uuid}` and include `memories.entry`, `memories.profile.setup`, `memories.timeline`, `memories.records.create`, `memories.records.show`, `memories.records.edit`, and `memories.media.show`.

Keep this story on the existing memory-product path unless implementation proves a project-wide translation path change is already required.
- `resources/views/components/layouts/memory.blade.php` already uses `__('memory.layout.*')` for memory layout navigation.
- Story 1.9 added localized timeline filter strings and tests for timeline filtering. Story 1.10 must preserve that filter behavior and complete localization coverage around it.
