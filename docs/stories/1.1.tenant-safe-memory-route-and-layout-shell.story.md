## Story

**As a** registered user,
**I want** to enter a dedicated memory product area for my workspace,
**so that** my memories are accessed through the correct tenant boundary and do not feel like an admin dashboard.

## Acceptance Criteria

1. Authenticated users can access a memory route group under `/memories/{tenant:uuid}`.
2. Unauthenticated visitors are redirected to authentication before reaching memory routes.
3. Users who are not members of the route tenant cannot access the memory area.
4. The memory area uses a dedicated authenticated layout, proposed as `resources/views/components/layouts/memory.blade.php`.
5. The layout includes workspace context, product navigation, profile or language access, and an account/dashboard escape path.
6. The layout does not include the public marketing footer, public hero treatment, Filament CRUD chrome, or table-first dashboard structure.

## Integration Verification

1. **IV1:** Existing Filament Dashboard routes for billing, account, users, roles, teams, and invitations still work.
2. **IV2:** Existing public marketing and authentication routes still render through their current layouts.
3. **IV3:** Feature tests cover authenticated access, unauthenticated redirect, and cross-tenant denial.

### Architecture Guidance

- The memory product should use `/memories/{tenant:uuid}` as the customer-facing route group while SaaSykit account operations stay in Filament Dashboard. [Source: docs/architecture.md#product-routing-and-ui-architecture]
- Recommended route shape uses Laravel named routes, route model binding, `auth`, `verified`, tenant membership middleware, and Livewire full-page components under the `memories.` name prefix. [Source: docs/ui-architecture.md#route-configuration]
- Middleware names in the architecture are placeholders; the rule is that tenant membership and profile completion are enforced through route-level/server-side checks, not Blade-only decisions. [Source: docs/ui-architecture.md#route-responsibilities]
- The route shell story's required test slice is guest redirect, member access, non-member denial, dedicated memory layout, and existing Filament/account route regression. [Source: docs/ui-architecture.md#required-coverage-by-story-type]

### Existing Codebase Context

- `routes/web.php` currently defines public, auth, checkout, blog, roadmap, invitation, invoice, and `/dashboard` routes; there is no memory route group yet. [Source: routes/web.php]
- `User::canAccessTenant(Model $tenant): bool` checks tenant membership through the existing `tenants()` relationship. [Source: app/Models/User.php]
- `Tenant::users()` is a `BelongsToMany` relationship using the existing `tenant_user` pivot model. [Source: app/Models/Tenant.php]
- Existing feature tests extend `Tests\Feature\FeatureTest`, which runs migrations/seeders, configures currency, disables exception handling and Vite, and provides `createUser()` and `createTenant()` helpers. [Source: tests/Feature/FeatureTest.php]

### UI and Layout Guidance

- `resources/views/components/layouts/memory.blade.php` is the customer-facing memory shell and must not place memory screens inside the public marketing layout or Filament Dashboard chrome. [Source: docs/ui-architecture.md#critical-coding-rules]
- The memory layout shell owns navigation, workspace context, profile/language access, and dashboard/account escape; individual screens should not duplicate this chrome. [Source: docs/ui-architecture.md#component-boundary-rules]
- The MVP layout should be authenticated, tenant-aware, compact, product-first, and free of public marketing footer or hero section. [Source: docs/architecture.md#product-routing-and-ui-architecture]
- Use custom Livewire/Blade product UI for memory workflows; Filament remains for billing, account, workspace settings, users, roles, teams, invitations, security, and admin/reporting. [Source: docs/ui-architecture.md#critical-coding-rules]
- Tailwind implementation should use existing utility conventions, prefer `gap-*`, visible focus treatment, subtle borders/minimal shadows, and stable responsive dimensions. [Source: docs/ui-architecture.md#tailwind-patterns]

### Livewire Guidance

- Use Livewire for server-driven memory screens and keep tenant state server-side. [Source: docs/ui-architecture.md#frontend-tech-stack]
- Existing architecture points memory Livewire classes to `app/Livewire/Memory` and views to `resources/views/livewire/memory`. [Source: docs/ui-architecture.md#primary-files-and-directories]
- Livewire 4 supports full-page components and class-based generation, but the developer should verify the local `make:livewire` options before generating files. [Source: .agents/skills/livewire-development/SKILL.md#creating-components]
- Livewire loops in future screens should use `wire:key`, loading states should use Livewire loading primitives, and actions must authorize/validate like HTTP requests; this story only needs a shell, so do not introduce premature client-side behavior. [Source: .agents/skills/livewire-development/SKILL.md#best-practices]
