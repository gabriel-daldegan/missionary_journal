## Story

**As a** workspace owner,
**I want** the memory workspace to use shared editing by default,
**so that** the owner and collaborator can both create and refine records during MVP validation.

## Acceptance Criteria

1. The system stores memory workspace settings scoped to `tenant_id`.
2. The default collaboration mode is `shared_editing`.
3. Owner and collaborator can both create and edit records under MVP `shared_editing`.
4. Future collaboration modes are represented at the workspace level, not as per-record missionary-vs-family visibility.
5. The MVP does not expose a collaboration mode selector unless a later story explicitly adds it.

## Integration Verification

1. **IV1:** Existing SaaSykit tenant membership, roles, and invitations remain the source of workspace membership.
2. **IV2:** Existing tenant plan/member limits remain usable for the two-user validation tier.
3. **IV3:** Policy or service tests verify owner/collaborator shared editing and cross-tenant denial.

## Implementation Interpretation

AC3 is satisfied in this story through service and policy-style authorization
decisions for future memory-record create/update actions. Do not create the
`memory_records` table, `MemoryRecord` model, record editor UI, or record CRUD
routes in Story 1.3; Story 1.4 owns diary record creation.

### Architecture Guidance

- The architecture starts with shared editing: owner and collaborator can both create and edit records inside the workspace. [Source: docs/architecture.md#core-architectural-principles]
- `Tenant` is the MVP workspace boundary; the owner is the tenant creator or existing admin role tenant user, and the collaborator is an invited tenant member with the existing user role. [Source: docs/architecture.md#tenant-and-access-architecture]
- Access should resolve the current tenant from the route or existing dashboard tenant context, reject users whose `User::canAccessTenant($tenant)` check fails, require `tenant_id` on workspace-owned memory tables, and use policy checks for list/view/create/update/delete/media access. [Source: docs/architecture.md#tenant-and-access-architecture]
- `memory_workspace_settings` stores memory-product settings for a tenant/workspace without changing the SaaSykit tenant model. It includes `id`, `tenant_id`, `collaboration_mode` defaulting to `shared_editing`, timestamps, and a unique index on `tenant_id`. [Source: docs/architecture.md#memory_workspace_settings]
- The MVP collaboration mode is `shared_editing`; future modes include `owner_review`, `read_only_collaborator`, and `custom_roles`, but the MVP should store only the default mode and enforce it through `MemoryRecordPolicy` and `MemoryWorkspaceSettingsService`. [Source: docs/architecture.md#collaboration-mode-architecture]
- `MemoryWorkspaceSettingsService` should resolve workspace settings, return the current collaboration mode, and centralize future mode changes. [Source: docs/architecture.md#service-architecture]
- Services own persistence and shared editing rules; middleware and policies own the primary access boundary for tenants, records, and media. [Source: docs/framework/coding-standards.md#component-boundaries]
