## Story

**As a** user,
**I want** a writing-first diary creation form,
**so that** I can preserve a meaningful experience while it is still fresh.

## Acceptance Criteria

1. Tenant members with completed memory profiles can open a diary create route.
2. The diary form requires body text and experience date.
3. Users can optionally add location, tags, and ordered Memory Highlights.
4. The form delegates persistence to `MemoryRecordService` and redirects after successful creation.
5. The form is writing-first and does not require technical metadata classification.
6. The form does not render missionary-only, family-visible, collaboration mode, public sharing, media upload, period/trip, detail/edit, AI classification, or email-source controls.
7. Records remain private to the tenant/workspace by default through route, middleware, policy, and service boundaries.

## Integration Verification

1. **IV1:** Story 1.4b service tests remain passing.
2. **IV2:** Non-members cannot access or submit the diary create flow for another tenant.
3. **IV3:** Livewire/feature tests cover successful creation, required validation, unsupported type behavior, highlights persistence through the service, and forbidden UI control absence.

## Implementation Interpretation

This is the third slice of PRD Story 1.4. It requires Story 1.4a and Story 1.4b. It adds only the diary creation route and Livewire UI. Do not add record detail, record editing, timeline grouping, filters, media upload/private delivery, period/trip creation, public sharing, reporting, inbound email automation, or per-record visibility controls.

After successful diary creation, redirect to `memories.timeline` or another existing memory route. Do not introduce the Story 1.5 record detail route just to satisfy this flow.

### Architecture Guidance

- Customer-facing memory routes live under `/memories/{tenant:uuid}` and use Livewire/Blade product UI, not Filament. [Source: docs/framework/tech-stack.md#memory-product-direction]
- Expected route hierarchy includes `/records/create/{type}`. [Source: docs/ui-architecture.md#route-hierarchy]
- `memories.records.create` is owned by `MemoryRecordEditor` and gated to tenant members with completed profile and create permission. [Source: docs/ui-architecture.md#route-responsibilities]
- Create actions should route explicitly to a supported record type; this story supports only `diary`. [Source: docs/ui-architecture.md#navigation-rules]
- Livewire actions validate input, authorize the action, delegate persistence, and redirect through named routes. [Source: docs/ui-architecture.md#livewire-action-pattern]
- Treat Livewire public properties as request-visible state; do not store secrets, full private paths, or authorization decisions in them. [Source: docs/framework/coding-standards.md#memory-product-rules]
- Memory Highlights are an optional ordered dynamic list with add, remove, edit, and accessible reorder controls where supported. [Source: docs/ui-architecture.md#critical-coding-rules]
