## Story

**As a** developer,
**I want** the tenant-scoped diary record schema, models, relationships, and factories in place,
**so that** diary creation can be implemented against stable persistence without reworking database foundations.

## Acceptance Criteria

1. The system has tenant-scoped persistence for memory records, memory tags, record-tag links, and Memory Highlights.
2. The schema supports diary records with `body`, `experience_date`, optional `location_name`, tags, and ordered highlights.
3. Period/trip fields may exist as nullable architecture-reserved fields, but no period/trip behavior is implemented.
4. Models, relationships, constants, and factories exist for record, tag, and highlight test setup.
5. The schema does not add missionary-only, family-visible, approval, public sharing, AI classification, or media behavior.

## Integration Verification

1. **IV1:** Migrations run cleanly with existing tenant, memory profile, and memory workspace settings migrations.
2. **IV2:** Existing memory routes and profile gate behavior remain unchanged.
3. **IV3:** Focused tests verify relationships, factories, tag tenant uniqueness, highlight ordering fields, and required tenant/author columns.

## Implementation Interpretation

This is the first slice of PRD Story 1.4. It creates only the database and model foundation needed by later diary creation stories. Do not create `MemoryRecordService`, `MemoryRecordPolicy`, `MemoryRecordEditor`, record creation routes, Livewire views, or user-facing record forms in Story 1.4a.

### Architecture Guidance

- `Tenant` is the MVP workspace boundary; memory-domain tables that belong to a workspace require `tenant_id`. [Source: docs/architecture.md#tenant-and-access-architecture]
- `memory_records` stores diary and trip/period records on the same tenant timeline. [Source: docs/architecture.md#memory_records]
- Diary records order by `experience_date`; `created_at` must not control primary timeline order. [Source: docs/architecture.md#memory_records]
- Diary records require `body` and `experience_date`; period records require period fields but are outside this story. [Source: docs/architecture.md#memory_records]
- `memory_tags` are tenant-scoped and use a unique index on `tenant_id` plus `slug`. [Source: docs/architecture.md#memory_tags-and-memory_record_tag]
- `memory_highlights` store one row per highlight item with `text` and `sort_order`; do not add AI classification fields. [Source: docs/architecture.md#memory_highlights]
