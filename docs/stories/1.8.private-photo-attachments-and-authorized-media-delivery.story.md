## Story

**As a** user,
**I want** photos attached to records to stay private,
**so that** sensitive memory media is protected by workspace access.

## Acceptance Criteria

1. Users can attach photos to diary records and trip/period records.
2. Photos remain connected to their parent memory record.
3. Photo upload respects validation limits of 2 GB per workspace, 10 MB per image, and 25 photos per record.
4. Memory photos use private storage and are not exposed through public media URLs.
5. Served image renditions are authorized by tenant/workspace access and parent-record visibility.
6. The implementation preserves a future S3-compatible storage path.

## Integration Verification

1. **IV1:** Existing public blog media behavior is not reused for private memory photos.
2. **IV2:** Users outside the tenant cannot access record media directly.
3. **IV3:** Tests cover authorized media access, unauthorized cross-tenant media denial, and attachment persistence.

## Implementation Interpretation

Story 1.8 adds photo attachment and private delivery to the existing memory record flow. Diary and period records already share the same tenant-scoped `MemoryRecord` model, editor, detail page, and timeline service after Story 1.7. This story should extend that boundary with private media behavior instead of creating a separate album, gallery, or public asset subsystem.

The preferred implementation is to make `MemoryRecord` a Spatie Media Library model with a `photos` collection stored on private storage, add upload state to the existing Livewire record editor, persist attachments through the memory service layer, and serve rendered images through an authorized `memories.media.show` route. Public blog media helpers such as `getFirstMediaUrl()` and public storage URLs must not be used for memory photos.

The scope is attachment and authorized display for existing diary and period create/detail/timeline surfaces. This story does not add filters, masonry layouts, maps, public sharing, per-record visibility toggles, album-generation language, reporting dashboards, or unrelated SaaS account/billing/workspace changes.

## Scope Boundaries

- Attach memory photos to `MemoryRecord` parents, not to tenant, user, blog, or standalone album records.
- Support photo upload from diary create and period create flows.
- Keep diary edit behavior stable; only include edit-time photo changes if they naturally fall out of the shared editor implementation and are covered by focused tests.
- Store memory photos on a private disk/path, using `config('memory.media.*')` or equivalent config values rather than direct `env()` access in app code.
- Keep the implementation compatible with future S3-compatible storage, but do not introduce external infrastructure or new dependencies without explicit approval.
- Serve images only through authorized Laravel routes or an equivalent private access mechanism.
- Do not expose original storage paths, filenames, public URLs, or private media metadata in Blade, Livewire public properties, logs, or JavaScript.
- Do not reuse public blog media display assumptions, public blog Blade components, public `public/storage` paths, or marketing/blog image patterns.
- Do not implement Story 1.9 timeline filters by date, tag, or location.
- Do not add public sharing, per-record visibility modes, masonry, maps, complex animation, album-generation copy, reporting dashboards, or account/dashboard/workspace-management changes.
